# Archive Oculus — 게임 제작 워크플로우 보강안 v1.0
*2026-06-19 — 기존 Step 1~6 파이프라인을 30분 분량 게임 제작용으로 보강*

---

## 배경

기존 파이프라인은 **유튜브 나레이션 대본** 집필 기준으로 설계됨.
게임(30분 분량, 분기형 비주얼 노벨) 제작에는 분기 설계와 화면 구현 단계가 빠져있어 보강 필요.

---

## 전체 구조 (보강 후)

```
Step 1   주제 선정 (기존 유지)
Step 2   딥 리서치 (기존 유지)
Step 3   설계 + 포맷 결정 (보강)
Step 3.5 서사 설계 + 분기 트리 설계 (보강)
Step 4   vn_script.md 집필 — 분기별 (보강)
Step 4.5 화면 프로토타입 — claude.ai 협업 (신규)
Step 5   game.html 빌드 — 포맷별 템플릿 (보강)
Step 6   OSMU 추출 (기존 유지)
```

---

## Step 2 — 딥 리서치 (변경 없음)

```
→ fact_sheet.md (역사 팩트 + 출처 URL)
→ 스탯 4~6개 설계 (신앙/평판/법적방어력/정신력 등)
→ 사용자 승인
```

## Step 3 — 설계 (보강)

### 기존
```
→ 지뢰 후보 6개 제안 → 사용자가 4개 선택
→ NPC 카드 3장 이상
→ scenario.json 골격 생성
→ 사용자 승인
```

### 추가 사항
```
포맷 결정 (필수, 이 단계에서 확정)
→ 기본형 / 법정물 / 미연시 / 육성 / 로그라이크 /
   추리물 / 탈출어드벤처 / 타이쿤 중 선택
→ archive_oculus_episode_formats_v1.md 참고
→ 포맷에 따라 Step 5 빌드 템플릿이 달라지므로 여기서 반드시 확정

지뢰 개수 조정
→ 30분 분량 기준 4개 → 6개로 확장 고려
   (스테이지 1~4 + 오프닝/엔딩 구조에 맞춰 조정)

고정 NPC 등장 여부 결정
→ archive_oculus_fixed_npc_cards_v1.md 참고
→ "이번 에피소드에 역병의사 등장하는가? 어떤 가면?"
→ "네피아 복선을 심을 단계인가?"
→ "글리치 NPC 후보가 될 만한 에피소드 NPC가 있는가?"
→ 등장 안 해도 무방 (모든 에피소드에 강제 X)

코미디 지뢰 1개 포함 (신규, 숏츠 공유/저장 지표 강화 목적)
→ 지뢰 6개 후보 중 최소 1개는 "황당하지만 실제 역사적 사실"
   기반의 코미디 지뢰로 설계 시도
→ Step 2 리서치 단계에서부터 fact_sheet.md에
   "황당한 실제 사례" 항목을 의도적으로 찾아볼 것
→ ⚠️ 전제 조건: 반드시 실제 역사적 근거가 있는 디테일만 사용
   (출처 URL 명시 가능한 것만 채택)
→ ⚠️ 해당 에피소드 리서치에서 적합한 실제 사례를 찾지 못하면
   코미디 지뢰는 과감히 패스. 허구로 지어내지 않음.
   (모든 에피소드에 코미디 지뢰가 강제되는 것이 아님 —
    "있으면 좋고, 없으면 그냥 진지한 지뢰 6개로 진행")
→ 톤 처리: 오르페아가 웃기려는 게 아니라
   오르페아는 평소처럼 냉소적인데 결과적으로 웃긴 상황이 되는 것
   (캐릭터 일관성 유지, 블랙코미디로 처리)
→ 배치: 4개 스테이지 중 1~2곳에만 (전부 코미디면 몰입감 깨짐)
→ 용도: 숏츠 "배드플레이형" 콘텐츠 전용 소재로 활용
   (archive_oculus_osmu_strategy_v2.md 섹션 6 참고)

예시 (세일럼 기준):
선택지 C "고양이가 범인이라고 우긴다" (코미디 지뢰)
→ 실제로 세일럼 재판에서 동물도 마녀 혐의로 기소된 사례가 역사적으로 존재
→ 오르페아: "...고양이가 마녀라고 주장하셨습니다.
   1692년 세일럼에서는 그것도 처형 사유입니다. 고양이도, 당신도."
```

## Step 3.5 — 서사 설계 (보강)

### 기존
```
→ 인과관계 설계 (파트2에서 심은 것 → 파트5에서 회수)
→ 7파트 개요 완성
→ scenario.json 잠금 🔒
→ 사용자 승인
```

### 추가 사항 — 분기 트리 설계

```
다이아몬드 구조 권장
(완전 자유 분기는 Step 4 집필량 폭발시킴)

스테이지마다 2~3개 선택지
→ 선택에 따라 스탯 변화는 다르지만
→ 중간 지점에서 다시 합류
→ 최종 스테이지(엔딩 직전)에서만 크게 갈림

예시 구조:
오프닝
  ↓
스테이지1 (선택 A/B/C) → 스탯 변화만 다름
  ↓ (합류)
스테이지2 (선택 A/B/C) → 스탯 변화만 다름
  ↓ (합류)
스테이지3 (선택 A/B/C) → 스탯 변화만 다름
  ↓ (합류)
최종 관문 → 누적 스탯에 따라 진짜 분기
  ↓
엔딩 A (퍼펙트) / 엔딩 B (일반생존) / 엔딩 C (사망) / 히든

엔딩 조건 명세 (필수)
→ 퍼펙트 생존: 구체적 스탯 임계값 (예: 전체 스탯 70 이상)
→ 일반 생존: 구체적 스탯 임계값
→ 사망: 특정 스탯 0 도달 또는 지뢰 3개 이상 적중
→ 히든: 특수 트리거 (예: 특정 스테이지에서 "아무것도 선택하지 않음")

scenario.json에 분기 트리 + 엔딩 조건까지 포함 후 잠금
```

## Step 4 — vn_script.md 집필 (보강)

### 기존
```
→ 씬 단위로 대사 + 선택지 + 스탯 변화 작성
→ 씬마다 사용자 승인
```

### 변경 사항
```
"씬 단위" → "분기 단위"로 전환

각 스테이지마다:
- 공통 도입부 (선택 전 상황 묘사) — 1회 작성
- 선택지 A/B/C 각각의 결과 텍스트 — 분기별 작성
- 합류 지점 텍스트 — 1회 작성

주의: 선택지마다 완전히 다른 장면을 쓰지 않는다
      대사 일부는 공유하고, 스탯 변화와 결과만 분기
      (집필량 2~3배 폭증 방지)

작업량 가이드:
기본형 4스테이지 + 다이아몬드 구조 기준
총 텍스트량은 기존 선형 대본의 약 1.5배 예상
```

## Step 4.5 — 화면 프로토타입 (신규)

```
주체: claude.ai (채팅 모드)

목적:
클코가 텍스트만 보고 빌드하면 UI 디테일을 놓칠 수 있음
화면으로 직접 보면서 다듬는 단계 필요

프로세스:
1. vn_script.md + scenario.json을 claude.ai로 전달
   (다운로드 후 채팅에 첨부 또는 복붙)
2. claude.ai가 React/HTML Artifact로 프로토타입 제작
   - 배경, 오르페아 등장, 타이핑 애니메이션,
     선택지 버튼, 스탯 UI까지 실제 작동하는 형태
3. 화면 보면서 톤/이펙트/레이아웃 조정
4. 확정되면 코드를 클코에게 전달

산출물: prototype.jsx 또는 prototype.html
        (Step 5에서 클코가 이를 기반으로 game.html 빌드)
```

## Step 5 — game.html 빌드 (보강)

### 기존
```
→ 브라우저에서 실행되는 단일 HTML 파일
```

### 추가 사항
```
포맷별 템플릿 분기 (Step 3에서 결정된 포맷 기준)

기본형: 기존 game.html 구조로 충분
법정물: 증거 패널 + 증언 반박 UI 컴포넌트 추가
미연시: 호감도 게이지 + 캐릭터 표정 전환 컴포넌트 추가
육성:   캘린더 + 다중 스탯 바 컴포넌트 추가
로그라이크: 죽음 카운터 + 회차별 정보 해금 로직 추가
추리물: 단서 노트 + 용의자 연결 맵 컴포넌트 추가
탈출어드벤처: 공간 맵 + 아이템 조합 패널 추가
타이쿤: 자원 현황판 + 거래 메뉴 추가

각 포맷 템플릿은 한 번 만들면 재사용 가능
(같은 포맷의 다음 에피소드는 데이터만 교체)

Step 4.5 프로토타입을 기반으로 최종 빌드
```
