## 1. 선택지 역할 셔플 규칙 (중요도 높음)

### 문제
현재 `stage_design.md`를 보면 함정 위치가 우연히도 매번 다르긴 하지만(스테이지1=A, 스테이지2=A, 스테이지3=A, 최종관문=A),
**실제로는 전부 A가 함정**이다. 패턴이 그대로 노출되어 있고, 이를 막는 셔플 규칙이 어느 스킬 파일에도 없다.

### 설계 규칙

```
대원칙 (2가지, 절대 충돌 없이 동시 적용):
  ① 위치 분산: 함정의 알파벳 위치는 스테이지마다 다르게 간다 (매번 다른 게 기본값,
     "2회 연속 같은 위치 금지" 같은 느슨한 기준이 아니라 한 에피소드 내 모든
     스테이지가 서로 다른 위치를 쓰는 것을 목표로 한다 — 스테이지 수가 알파벳
     종류보다 많아 부득이 겹칠 때만 "연속 금지"로 완화)
  ② 매력도 불변: 위치가 어디로 가든, 함정 선택지의 "그럴듯함(현대 상식상 합리성)"은
     절대 낮아지지 않는다. 위치를 옮긴다고 문장을 약하게 쓰면 셔플의 의미가 없다.
     함정은 항상 "내가 봐도 이게 정답 같은데?" 수준을 유지해야 하고,
     이건 위치 설계와 별개로 항상 100% 지켜야 하는 문장 작성 기준이다.

EP.01 세일럼 — 함정 위치 순환표 (수정안 — ①번 원칙대로 전부 다른 위치):
| 단계      | 함정 위치 | 탈출구 위치 | 비고                          |
|-----------|-----------|-------------|-------------------------------|
| 스테이지1 | A         | B           | C는 소극적 생존(무위험 저보상) |
| 스테이지2 | C         | B           | A는 가장 강력한 함정(역사적 함정으로 표시되어 있던 자리 — 위치만 C로 이동, "결백 주장" 문장의 설득력은 그대로 유지) |
| 스테이지3 | B         | A 또는 C    | 함정 위치 변경에 따라 선택지 텍스트 자체도 B자리에 맞게 재배치 필요 |
| 최종관문  | A         | B, C        | 4단계 중 가장 마지막이라 A 재사용 가능 (직전 스테이지가 B였으므로 연속 아님) |

⚠️ 기존 stage_design.md는 4단계 전부 함정이 A — 이게 클코가 지적한 핵심 문제.
   위 표로 교체 시 스테이지2/3은 선택지 A/B/C 텍스트를 함정 위치에 맞게
   다시 배치해야 하며(문장 내용은 유지, 슬롯만 이동), 이 작업은 Step 3.5 재승인 대상.

규칙화 (향후 에피소드 공통 — 모든 에피소드에 일괄 적용):
1. 한 에피소드 내 스테이지 수가 3~4개일 때는 함정 위치가 전부 달라야 한다
   (스테이지 수가 선택지 알파벳 수(보통 3~4개)를 넘지 않는 한 100% 분산 가능).
2. 선택지 "텍스트 매력도"는 위치와 완전히 독립적으로 설계한다 — 함정 문장을 쓸 때
   "이게 A자리니까 더 그럴듯해야 하나"같은 위치 종속적 사고를 하지 않는다.
   매 함정은 항상 최고 수준의 설득력으로 쓴다, 위치는 그 다음에 배정한다
   (순서: 문장 먼저 최고 수준으로 작성 → 위치는 분산 규칙에 따라 나중에 배정).
3. scenario.json의 각 stage.choices 배열 순서 자체를 셔플 대상으로 하지 말고,
   "trap_choice_id" 필드를 별도로 둬서 어떤 id가 함정인지 명시한다.
   (배열 순서까지 섞으면 집필 시 오르페아 평가 순서가 꼬일 수 있음)

scenario.json 추가 필드 제안:
{
  "stages": [
    {
      "id": 1,
      "trap_choice_id": "A",
      "escape_choice_ids": ["B"],
      "neutral_choice_ids": ["C"]
    }
  ]
}
```

## 2. 스탯 임계값 체계 (중요도 높음)

### 문제
`stage_design.md`에 "정신력 ≤ 30이면 증상 강화"라는 1회성 언급만 있고,
어느 스탯이 얼마 이하일 때 무엇이 일어나는지 전체 체계가 없다.

### 설계 규칙

```
모든 스탯에 3단계 임계값을 둔다: 경고(Warning) / 위험(Critical) / 봉괴(Collapse)
(Archive_Oculus_UI_Kit_v1.md의 "안정/위험/위기/사망 직전" 4단계 색상 체계와 정합)

공통 구조:
  100~61: 안정 — 평상시 선택지 전부 노출
  60~31:  경고 — 일부 선택지에 페널티 텍스트 추가 표시, 오르페아 코멘트 톤 변화
  30~11:  위험 — 특정 선택지 비활성 또는 강제 발동, 연출 강화(환각/떨림 등)
  10~0:   봉괴 — 해당 스탯발 자동 배드엔딩 트리거

EP.01 세일럼 — 스탯별 임계값 표:

| 스탯           | 경고(60↓)              | 위험(30↓)                          | 봉괴(0)                  |
|----------------|-------------------------|--------------------------------------|---------------------------|
| 신앙 (Faith)   | 오르페아가 "신앙심이 옅어지는군요" 식 1회 언급 | 예배/신앙 관련 선택지에 의심 가중치 추가 표시 | 공동체 추방 → 사망 엔딩 경로 강제 진입 |
| 평판 (Reputation) | 메리 등 NPC 호의 반응 약화 | 증인 신뢰도 하락 — 최종관문 B 선택지 생존률 하향 조정 | 이웃 전원 적대 → 탈출/조력 선택지 전부 비활성 |
| 법적방어력 (Defense) | 심문 단계 선택지에 경고 문구 추가 | 최종관문 A(결백 주장) 선택지가 자동으로 최하 생존률로 고정 표시 | 법정 발언권 박탈 — 최종관문 진입 시 자동 패소 안내 |
| 정신력 (Sanity) | 스테이지2 환각 연출 1단계 강화 (기존 설계) | 스테이지2 환각 연출 2단계 강화 + 자백계 선택지(C) 생존률 표시가 흐려짐(판단력 저하 표현) | 자백 강요 선택지 자동 발동 — 플레이어 선택 무시하고 최악 분기로 강제 진행 |

규칙화 (향후 에피소드 공통):
1. 모든 에피소드는 최소 1개 스탯에 "봉괴 시 선택지 강제 발동/비활성" 메커니즘을 가져야 한다.
   (단순 수치 표시만으로는 게임적 긴장감이 약함)
2. "경고" 단계는 텍스트 변화만, "위험" 단계부터 실제 선택지 구조에 개입한다.
3. scenario.json에 stat별 threshold 필드 명시:

{
  "stats": [
    {
      "id": "sanity",
      "name": "정신력",
      "max": 100,
      "initial": 100,
      "thresholds": {
        "warning": 60,
        "critical": 30,
        "collapse": 0
      },
      "collapse_effect": "스테이지2 C선택지 자동 강제 발동"
    }
  ]
}
```

## 3. NPC 호감도(affection) 추적 방식 (중요도 높음)

### 문제
초기 scenario.json 골격(step3.md)에는 `affection` 개념이 있었으나 `stage_design.md`/`npc_cards.md`에서는
"스탯 기반"인지 "선택지 기반"인지 불명확하게 사라졌다. 메리의 배신이 무엇으로 결정되는지 모호함.

### 설계 규칙

```
원칙: 호감도는 별도 추적하지 않고 "선택지 기반 플래그"로 단순화한다.
(이유: 세일럼 에피소드는 3스테이지 단편이라 별도 affection 수치를 둘 만큼
 NPC 접촉 빈도가 많지 않음. 수치화하면 오히려 설계 부채만 늘어남.
 단, 장기 에피소드/육성형 포맷에서는 affection 수치 부활 고려 — 아래 "확장 시" 참고)

EP.01 적용 — 메리 배신 조건 명시:
  현재 npc_cards.md 기술: "스테이지1에서 플레이어가 A 선택(탄원서 제출) 시
                            최종관문에서 반드시 지목"
  → 이는 이미 "선택지 기반" 트리거다. 수치(affection) 불필요.
  → scenario.json에 flag 필드로 명시:

{
  "npcs": [
    {
      "id": "mary",
      "betrayal_trigger": {
        "type": "choice_flag",
        "condition": "stage1_choice == 'A'",
        "result": "최종관문에서 플레이어 지목 증언"
      }
    }
  ]
}

세라(ally) 비교 — 세라는 배신 없음, 조건부 등장만:
{
  "id": "sara",
  "appearance_trigger": {
    "type": "always",
    "info_unlock_condition": "stage3_choice == 'B'"
  }
}

규칙화 (향후 에피소드 공통 — 모든 에피소드에 일괄 적용):
1. 기본형(단편, 3~4스테이지): NPC 관계는 "선택지 기반 플래그"로만 추적한다.
   - betrayal_trigger / appearance_trigger / unlock_condition 세 가지 타입으로 충분.
2. 확장형(미연시·육성형 포맷 — archive_oculus_episode_formats_v1.md 참고):
   호감도 게이지가 필요한 포맷에서만 0~100 수치형 affection 부활.
   이 경우 scenario.json에 stats와 동일한 구조로 분리:
   "npc_affection": [{"npc_id": "...", "initial": 50, "max": 100}]
3. **[범용 규칙]** "도덕적 무게가 실리는 엔딩에서 누군가 희생되거나 지목된다"는
   서사 구조를 쓰는 모든 에피소드는, 그 희생자를 이름 없는 단역이 아니라
   **반드시 npc_cards에 이미 등장한 "페이크 조력자(달콤해 보이지만 함정 역할)" 캐릭터로 고정한다.**
   - 근거: npc_card_template.md 자체가 "달콤해 보이는 캐릭터가 함정" 구조를 표준으로 못박고 있음.
     이 캐릭터는 보통 극 중반에 등장해 복선(배신 신호)을 갖고 있으므로,
     엔딩에서 그 복선을 회수하는 것이 자연스러운 기본값이다.
   - 에피소드별로 "누구"는 다르다 (EP.01=메리, 다른 에피소드=각 에피소드의 페이크 조력자).
     "메리"라는 이름이 고정되는 게 아니라 **"페이크 조력자 역할이 지목 대상이 된다"는
     구조 자체가 고정된다.**
   - scenario.json 필드로 일반화:
     {"npcs": [{"id": "...", "role": "fake_ally", "sacrifice_eligible": true}]}
     → 엔딩 설계 시 sacrifice_eligible: true인 NPC를 자동으로 지목 후보 1순위로 채택.
   - 예외: 해당 에피소드에 페이크 조력자 캐릭터 자체가 없는 포맷(예: 단독 생존,
     NPC 상호작용 최소형)이라면, 그때만 이름 없는 단역으로 대체한다 — 이게 fallback이지 기본값이 아니다.
4. 일반생존류 엔딩의 "동료 지목/희생" 텍스트는 위 3번 규칙에 따라
   매 에피소드 자동으로 특정 NPC ID와 연결되며, 모호하게 남겨두지 않는다.
```

## 4. 엔딩 씬 설계 (중요도 높음)

### 문제
퍼펙트/일반생존/사망 엔딩 각각 어떤 장면인지 미설계. 특히 일반생존 엔딩에서
"이웃 1명 공범 지목"이 누구인지, 그 도덕적 무게를 어떻게 다룰지가 핵심 공백.

### 설계 규칙

```
엔딩 3종 모두 아래 4개 필드를 채워야 "엔딩 설계 완료"로 간주한다:
  - trigger_condition (이미 stage_design.md에 있음 — 재확인용)
  - scene_beat (장면에 반드시 들어가야 할 핵심 비트, 대사 아님 — 지시문)
  - npc_resolution (관련 NPC들이 각각 어떻게 처리되는지)
  - orphea_tone (이 엔딩에서 오르페아 톤이 평소와 어떻게 다른지/같은지)

--- 퍼펙트 엔딩 ---
trigger_condition: 스테이지1 B → land_deed_info 보유 + 최종관문 C 선택
scene_beat:
  - 코윈이 토지 양도 서류 앞에서 말을 잃는 순간 (npc_cards.md 기존 시그니처 대사 활용:
    "...이 서류의 날짜가 기소일 이전이라고요.")
  - 플레이어는 법정에서 풀려나지만 마을은 그대로 — "이겼다"가 아니라 "빠져나왔다"는 톤 유지
  - 메리/세라 언급 여부: 메리는 스테이지1 B 선택 경로이므로 배신 트리거 미발동 →
    엔딩에서 메리는 등장하지 않거나 짧게 "여전히 마을에 남은 이웃" 정도로만 처리
npc_resolution:
  - 코윈: 무력화, 침묵 (패배/무력화 조건 그대로)
  - 아비게일: 그대로 발작 지속 (npc_cards.md: "패배 조건 없음, 마을을 떠남"과 연결해
    엔딩 에필로그 자막/내레이션에서 "이후 역사" 식으로 1줄 처리)
orphea_tone: harness.md 최대 감정 표현 기준 그대로 사용 —
  "기록할 가치가 있군요" 수준. 단, 흥분 없이 우아하게.

--- 일반생존 엔딩 (핵심 보강 지점) ---
trigger_condition: 스테이지3 B → confession_strategy 보유 + 최종관문 B 선택
scene_beat:
  - 플레이어는 자백하되 "이미 죽은 사람 이름"을 댄다는 세라의 조언(npc_cards.md
    시그니처 대사 "이미 죽은 사람 이름으로")을 따랐다면 도덕적 부담이 줄어드는 경로,
    이를 따르지 않았다면 산 사람을 지목하는 경로 — 두 갈래로 세분화 가능 (선택, 향후 검토)
  - 지목 대상: [3번 항목 범용 규칙] 적용 — 에피소드의 "페이크 조력자" 역할 NPC를
    자동으로 지목 후보 1순위로 채택한다. EP.01에서는 메리(npc_cards.md상 "페이크 조력자"
    role 명시됨)가 해당. 다른 에피소드는 각자의 페이크 조력자 NPC로 자동 치환된다 —
    "메리"라는 이름이 아니라 **"role: fake_ally" 태그가 달린 NPC가 지목 대상이 된다는
    구조 자체가 모든 에피소드에 공통 적용**.
  - 플레이어는 풀려나지만 자유의 몸으로 마을을 떠나는 모습 + 지목된 인물이
    끌려가는 모습이 동시에 그려지는 구조 (둘 다 보여주되 플레이어의 죄책감을
    오르페아가 대신 언어화하지 않고, 침묵 또는 단문으로 처리)
npc_resolution:
  - 페이크 조력자(EP.01=메리): 공범으로 지목되어 구금. npc_cards.md의 기존
    "배신/반전 시" 대사는 그 NPC가 증인으로 나오는 경로에서 쓰고,
    플레이어가 직접 지목하는 이 엔딩에서는 해당 NPC의 반응을 대사 없이
    시선/침묵으로 처리하는 것을 표준으로 한다 (대사보다 침묵이 더 무거움 — 전 에피소드 공통 연출 원칙)
orphea_tone: ⚠️ 시리즈 전체에서 유일하게 독설 톤을 거두는 지점으로 설계 권장.
  harness.md의 "절대 없음: 분노, 흥분, 안타까움" 규칙은 유지하되,
  이 엔딩에서만 평소보다 짧고 건조한 대사 1줄로 끝내는 것을 표준화:
  예시 방향성(대사 확정 아님, Step4 집필 시 orphea_voice_guide.md 톤으로 재작성):
  "...생존하셨습니다. 그게 전부입니다." 정도의 절제된 톤.
  → 풍자나 비유 없이 가장 짧은 문장으로 처리 = 이 엔딩 타입 전체의 시그니처
    (에피소드마다 같은 구조의 엔딩이 나오면 항상 이 절제된 톤 적용).

--- 사망 엔딩 ---
trigger_condition: 최종관문 A 선택 OR 특정 스탯 0 도달 OR 누적 지뢰 과다 적중
scene_beat:
  - 트리거 경로가 3가지이므로 각각 다른 사망 비트 필요 (현재 1개로 뭉뚱그려짐):
    1) 최종관문 A 선택 경로: 법정에서 교수형 선고 — npc_cards.md 코윈 시그니처 대사 활용
    2) 스탯 봉괴 경로(예: 정신력 0): 자백 강요 강제 발동 후 공범 다수 지목 →
       오히려 "스스로 더 큰 죄를 만든" 버전의 죽음 (1번과 톤 차별화 필요)
    3) 누적 지뢰 과다 적중: 특정 단일 사건이 아니라 "여러 차례의 작은 실수가 누적"된
       죽음 — 엔딩 직전 짧은 몽타주식 회상 (이전 스테이지 선택 3개를 1줄씩 회수)
  - Death Screen UI 그대로 사용 (Archive_Oculus_UI_Kit_v1.md):
    "ARCHIVE FAILED / Historical Record Terminated"
npc_resolution:
  - 아비게일: 침묵 없이 계속 발작 — 죽음에 책임 없는 듯한 태도 유지 (동기상 당연함)
  - 코윈: 토지 몰수 성공 — 엔딩 후 1줄로 "그의 동기가 달성되었다"는 사실만 차갑게 처리
orphea_tone: harness.md 기존 Death Screen 대사 기준 유지:
  "기록 저장 완료. 유감스럽게도 시체 상태로군요." — 수정 불필요, 그대로 표준.

규칙화 (향후 에피소드 공통):
1. 모든 엔딩은 trigger_condition / scene_beat / npc_resolution / orphea_tone
   4필드를 stage_design.md 또는 별도 endings.md에 채워야 Step 3.5 잠금 가능.
2. 엔딩이 여러 트리거 경로를 가지면(사망 엔딩처럼), 경로별로 scene_beat을 분리한다.
   "하나의 엔딩 = 하나의 장면"이 아니라 "하나의 엔딩 ID = 여러 변주"가 기본값.
3. 도덕적 무게가 실리는 엔딩(생존을 위해 누군가를 희생시키는 구조)은
   오르페아 톤을 의도적으로 이 엔딩에서만 다르게 설계해 임팩트를 만든다.
```

## 5. 오르페아 코멘트 타이밍 (중요도 중간)

### 결론
**기존 harness.md / 최종_대본_프롬프트.txt의 선택지 포맷을 그대로 표준으로 명문화.**
별도 타이밍 레이어를 추가하지 않는다 (산문 집필량 증가 방지).

```
표준 타이밍 (이미 확정된 포맷, production_spec에 "명문화"만 하면 됨):

1) 선택지 제시 시점에 각 선택지마다 1회씩 — "예상 생존률 + 오르페아 평가"
   (harness.md [선택지 포맷] 그대로)
2) 선택 후 결과 공개 직전 — 전환 멘트 1회
   ("선택은 끝났습니다. 이제 실제 역사 속 사람들이 어떤 결정을 내렸는지...")
3) 결과 공개 직후 — 배드/굿 엔딩 여부에 따라 오르페아 짧은 코멘트 1회
   (이 부분이 현재 가장 명문화가 약한 지점 — 아래 추가)

production_spec에 추가할 명시 규칙:
"오르페아는 선택지당 1회(제시 시), 전환 시 1회, 결과 공개 직후 1회,
 총 스테이지당 최대 (선택지 수 + 2)회 발화한다.
 결과 공개 직후 코멘트는 harness.md [오르페아 목소리 공식]을 따르되,
 절대 '잘했다/틀렸다'류 평가어를 쓰지 않고 사료적 사실의 비틀기로 대체한다."

→ 이건 새 타이밍을 만드는 게 아니라 기존 3박자(제시/전환/결과직후)를
   문서로 명문화만 하면 됨. harness.md 159번 줄 [선택지 포맷] 섹션 끝에
   "결과 공개 직후 오르페아 1줄 코멘트 필수" 조항만 추가 권장.
```

## 6. 훅 선택지 위치 설계 (중요도 중간)

### 설계 규칙

```
"훅 선택지"란: 숏츠 타입A(클리프행어)/타입B(배드플레이) 소재로 쓰일,
              특히 임팩트가 큰 단일 선택지를 가리킴
              (archive_oculus_osmu_strategy_v2.md 섹션 6 숏츠 전략과 연결)

배치 규칙:
1. 타입A(클리프행어)용 훅 = 가장 긴장감 높은 스테이지의 함정 선택지
   EP.01 기준: 최종관문 A ("결백합니다. 물리적 증거를 요구합니다")
   - 사망 엔딩 직결 + 코윈/아비게일 동시 등장이라는 정보 밀도 → 훅 가치 최고
2. 타입B(배드플레이)용 훅 = 코미디 지뢰가 있는 스테이지 우선,
   없으면 "현대인이 가장 많이 낚일 것 같은" 함정 선택지
   EP.01 기준: 스테이지2 A ("하나님 앞에 맹세코, 저는 결백합니다")
   - "결백 주장 = 사형"이라는 반직관성이 가장 강한 인지부조화 포인트
3. 한 에피소드에 훅 선택지는 최대 2개(타입A 1개 + 타입B 1개)만 지정한다.
   모든 함정을 훅으로 쓰면 숏츠 차별화가 안 됨.

scenario.json 추가 필드:
{
  "stages": [
    {
      "id": "final_gate",
      "hook_tag": "cliffhanger"
    },
    {
      "id": 2,
      "choices": [
        {"id": "A", "hook_tag": "badplay"}
      ]
    }
  ]
}

규칙화 (향후 에피소드 공통):
- 훅 지정은 Step 3.5(서사 설계) 단계에서 stage_design 확정과 동시에 결정한다.
- 매 에피소드 동일한 스테이지 번호에 훅을 고정하지 않는다
  (예: "항상 최종관문이 훅"이면 그것도 패턴 학습 위험 — 셔플 원칙과 동일 논리 적용).
```

## 7. BGM 전환 큐포인트 + 이미지 에셋 리스트 (중요도 중간)

### 방향 전환
당초 "스테이지 전환 시점에 큐포인트를 미리 설계"하는 방식으로 접근했으나,
디렉터 피드백에 따라 **방식을 바꾼다**: 큐포인트/에셋을 대본 작성 *전*에 미리 정하지 않고,
**대본(Step 4) 완성 후 그 텍스트를 분석해서 자동으로 리스트를 뽑아내는 단계**로 전환한다.
이유: 어떤 장면에 어떤 음악/이미지가 필요한지는 실제로 써봐야 정확히 보인다.
미리 정하면 대본이 그 틀에 맞춰 깎이고, 막상 써보면 안 맞는 경우가 생긴다.

### 설계 규칙 — extractor.md에 신규 출력 2종 추가

```
현재 extractor.md(이미 존재하는 OSMU 자동 추출 에이전트)는
script.md + scenario.json을 읽어 shorts_hooks.json / web_article.md를 생성한다.
같은 방식으로 아래 2개를 추가 출력시킨다.
(참고: "visual_prompts.md"는 하네스 필터 버전1 문서에 이미 언급되어 있었으나
 extractor.md 실제 출력 목록에는 빠져 있었음 — 이번에 정식 편입)

--- 신규 출력 4: visual_prompts.md ---
대본을 장면 단위로 훑어, 이미지 생성이 필요한 지점을 자동 추출.

추출 기준 (대본에서 다음 신호를 찾는다):
  - 새 NPC 최초 등장 장면 (npc_cards.md의 "행동 버릇 — 연출용" 항목과 매칭)
  - 스테이지 전환 시 배경이 바뀌는 지점 (마을→감옥→법정 등 공간 전환)
  - 엔딩 직전 클라이맥스 비주얼 (예: 최종관문 폭로 순간)
  - UI 키트에 정의된 고정 화면 (오프닝/Death Screen/Success Screen 등은
    Archive_Oculus_UI_Kit_v1.md 프롬프트 그대로 재사용, 매번 새로 만들지 않음)

각 항목은 아래 형식으로 출력:
{
  "scene_id": "stage1_intro",
  "source_line": "[대본에서 해당 장면을 가리키는 문장 1줄 인용 또는 위치 설명]",
  "image_type": "background | npc_portrait | key_visual",
  "prompt": "[Archive_Oculus_UI_Kit_v1.md의 Master Style Prompt 톤 유지한 영문 프롬프트]"
}

--- 신규 출력 5: bgm_cuesheet.json ---
대본을 텐션 곡선 기준으로 훑어, 음악 전환이 필요한 지점을 자동 추출.

추출 기준 (대본에서 다음 신호를 찾는다):
  - 오프닝 시작 / 낭만 파괴 지점 (파트1 "30초 전복적 후킹" 경계)
  - 각 스테이지 진입 시점 (공간/긴장감 변화)
  - 선택지 제시 직전 (긴장 고조 시작점)
  - 스탯 위험/봉괴 임계값 도달 지점 (2번 항목 스탯 임계값과 연동 —
    해당 지점이 대본에 등장하면 자동으로 큐포인트 후보가 됨)
  - 엔딩 분기 직후 (3종 엔딩 각각 다른 큐 — 4번 항목 orphea_tone과 연동)

각 항목은 아래 형식으로 출력 (트랙 ID는 비워두고 "성격"만 채워서 production_spec에서
실제 트랙과 매칭하도록 함 — extractor가 트랙명을 지어내지 않는다):
{
  "cue_id": "stage2_entry",
  "source_line": "[대본에서 해당 지점을 가리키는 문장 1줄 인용 또는 위치 설명]",
  "transition_type": "fade | hard_cut | layer_add",
  "intended_mood": "[예: 폐쇄적, 불협화음, 절제된 안도 등 — 4/2번 항목 톤과 정합]",
  "track_id": null
}
```

### extractor.md 실행 순서 갱신 (제안)

```
1. script.md 전체 읽기
2. scenario.json 읽기
3. shorts_hooks.json 생성 (기존)
4. web_article.md 생성 (기존)
5. visual_prompts.md 생성 (신규)
6. bgm_cuesheet.json 생성 (신규)
7. scenario.json 최종 업데이트 (기존)
8. 완료 보고: 생성된 파일 목록 + 각 파일 요약
```

### 규칙화 (향후 에피소드 공통 — 모든 에피소드에 일괄 적용)
```
1. 이미지/BGM 리스트는 절대 대본 집필(Step 4) *이전*에 확정하지 않는다.
   순서는 항상: 대본 완성 → extractor 실행 → 리스트 산출 → 실제 에셋 제작.
2. extractor는 "어떤 지점에 무엇이 필요한지"와 "어떤 분위기인지"까지만 정하고,
   실제 트랙 선곡/이미지 최종 생성은 production_spec 단계(클코 또는 별도 작업)에서
   진행한다 — extractor의 역할은 자동 스카우팅이지 최종 제작이 아니다.
3. 위 추출 기준 항목들(스테이지 전환, 스탯 임계값, 엔딩별 톤 등)은
   이 보강 문서의 2번/4번 항목과 1:1로 연동되므로, 2번/4번 설계가
   바뀌면 extractor의 추출 기준도 같이 갱신해야 한다.
```
