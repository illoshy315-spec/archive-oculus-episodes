## 0. 핵심 방향 전환

```
기존 목표: 블로그(정보 전달, 일방향)

이유:

→ 블로그보다 커뮤니티가 이 설계를 훨씬 잘 활용함
```

## 1. 채널별 콘텐츠 형태 (확정)

```
유튜브 (롱폼)
게임 플레이 영상 편집본 + 나레이션, 전체 스토리 공개

틱톡/숏츠
분기점(선택의 순간) 위주 편집
⚠️ 클리프행어 원칙: 선택 직전 ~ 오르페아 코멘트까지만 보여주고
   결과는 보여주지 않음 ("당신이라면? 👉 게임에서 확인")
   결과까지 보여주면 게임 유입 동기가 사라짐

인스타/스레드 (카드뉴스)
역사적 맥락/상황 설명. 다크 아카데미아 비주얼 큐레이션
정보량: fact_sheet.md의 일부만 (자극적 핵심만, 휘발성)
관점: 1인칭 몰입형 ("당신이 그 시대에 있었다면")
마지막 장: 게임 유입 CTA + 오르페아 캐릭터 노출 (떡밥 겸용)

웹 — 커뮤니티 + 아티클 (역할 변경)
단순 블로그가 아니라 토론/제보가 쌓이는 공간으로 설계
세부는 4번 항목 참고
```

## 2. 콘텐츠 중복 방지 — "채택되지 않은 기록" 전략

### 문제
카드뉴스와 웹 아티클이 같은 fact_sheet.md를 쓰면 콘텐츠가 겹침
(알고리즘 중복 패널티 + 유저가 "또 같은 내용"이라고 느낌)

### 해결책 — 정보량/관점 차등화 + 새로운 콘텐츠 축 추가

```
1차 해법: 가공 방식 차별화
카드뉴스 = 맛보기 + 임팩트 (fact_sheet.md의 일부)
웹 아티클 = 정찬 + 깊이 (fact_sheet.md 전체 + 추가 분석)

2차 해법 (★ 더 강력함): "채택되지 않은 기록" 콘텐츠 축 신설
Step 3에서 지뢰 후보 6개 중 4개만 채택하고 2개는 버려짐
→ 이 버려진 2개가 지금까지 콘텐츠로 전혀 안 쓰이고 있었음
→ 이걸 웹 아티클 전용 콘텐츠로 활용

형식: "세일럼 마녀재판: 게임에 담지 못한 기록들"
내용: 지뢰로 채택 안 된 역사적 사실 1개당 1섹션
      - 왜 이게 지뢰가 될 수 있었는지
      - 당시엔 왜 이런 일이 가능했는지 (사회/문화적 맥락)
      - "만약 게임에 있었다면 어떤 선택지였을까" (재미 요소)

효과:
게임/카드뉴스/숏츠 = "채택된 이야기" (압축, 핵심)
웹 아티클 = "채택되지 않은 이야기" (확장, 깊이)
→ 같은 소스에서 나왔지만 완전히 다른 콘텐츠
→ 게임 해본 사람도 "오 이것도 있었어?" 새로운 정보로 느낌
```

## 3. 클코 파이프라인 연동 — Step 2/3 산출물 추가 활용

```
Step 2 완료 시점 (게임 제작 시작 전)
→ fact_sheet.md 완성 직후 카드뉴스 제작 착수 가능

→ 채택 안 된 지뢰 후보 2개를
  "채택되지 않은 기록" 웹 아티클 재료로 자동 보관
→ 클코 워크플로우에 "미채택 지뢰 보관" 단계 추가 권장
```

## 4. 웹 = 커뮤니티 설계

### 콘텐츠 엔진은 이미 세계관에 내장되어 있음

```
ARG 복선 시스템      → "이거 발견했는데 너네도 봤어?" 제보 스레드
오르페아 vs 네피아   → 진영 논쟁 스레드 자연 발생
역병의사 정체        → 추리/미스터리 게시판
글리치 NPC 목격담    → "나도 3화에서 봤어" 제보형 재방문 유도
채택되지 않은 기록   → "만약 이것도 지뢰였다면?" 유저 참여형 가정 스레드
```

### 플랫폼 선택

```
초반: 기존 플랫폼 활용 (디스코드 + 레딧 서브레딧 등)
      → 진입장벽 낮음, 초기 트래픽 모으기 쉬움
팬덤 형성 후: 자체 웹 커뮤니티로 이전/병행
      → 브랜드 완전 소유
```

### 첫 떡밥 설계 원칙

```
EP.01부터 "토론거리 1개"를 의도적으로 명확히 심을 것
너무 어렵지도 너무 쉽지도 않은 난이도
예: "역병의사가 왜 거기 있었을까?"
```

## 5. 브랜드 보이스 — 오르페아 페르소나 (전 채널 통일, 확정)

### 원칙

```
"Archive Oculus 공식" 계정이 오르페아 캐릭터 보이스로 운영
(브랜드 페르소나 마케팅 기법 — 공식임을 명시한 채널에서만 적용)

⚠️ 절대 금지: 기존 커뮤니티(레딧 등)에 정체를 숨기고
   오르페아인 척 잠입 활동하는 것 — 자작극(astroturfing)에 해당,
   발각 시 브랜드 신뢰도에 치명타. 공식 채널에서 투명하게 운영할 것.
```

### 적용 범위

```
유튜브 — 영상 나레이션
인스타/스레드 — 카드뉴스 캡션, 댓글 반응
디스코드/커뮤니티 — 공지, 운영 멘트
웹 아티클 — 본문 톤

전 채널 동일한 오르페아 보이스로 일관성 유지
→ 어느 채널에서 만나도 "같은 세계"라는 느낌
```

### 보이스 예시

```
카드뉴스 캡션:
"1692년, 세일럼. 의심받는 데 이유는 필요 없었습니다. —기록 보관자, 오르페아"

댓글 반응:
"흥미로운 질문이군요. 답은... 기록되어 있습니다. 직접 찾아보시길."

업데이트 공지:
"새로운 기록이 추가되었습니다. 이번엔 누가 살아남을까요. 유감스럽게도, 기대는 안 합니다."

디스코드 환영 메시지:
"또 한 명의 관찰자가 접속했군요. 기록을 즐기시길. 살아남는 건 당신 몫입니다."
```

### 예외 — CS/실무 응대는 톤 분리 (확정: 네피아 담당)

```
오르페아 보이스는 "콘텐츠성 소통"에만 적용
(공지, 캡션, 가벼운 댓글, 떡밥)

실무성 소통(버그 신고, 결제 문의, 진지한 불만)은
네피아 보이스로 전환 — 친절하되 기대 없는 톤으로 명확하게 처리
→ 캐릭터 일관성 유지하면서도 실무 응대 명확성 확보
→ 부가 효과: "버그 신고하면 네피아가 받는다"는 것 자체가
   재미 요소로 작동할 수 있음 (단, 실제 응대 품질은 진지하게 유지)
```

### 8-1. 플랫폼 전략 — 웹 우선 (신규 확정)

```
결론: 앱스토어 출시보다 웹(브라우저) 배포를 1단계로 우선한다

이유 1 — 앱스토어 심사 마찰
애플/구글 심사 대기(수일~수주) + 반려 리스크
→ EP 단위로 빠르게 검증/발행해야 하는 제작 주기와 충돌
→ 초반 단계(엔진/반응 테스트)에는 부적합

이유 2 — 서버비 부담 오해 해소
비주얼노벨류는 실시간 DB 조회가 필요한 구조가 아님
선택지/스탯/분기는 클라이언트(브라우저) 단에서 JS로 처리
→ 정적 호스팅(GitHub Pages, Netlify, Vercel 등)으로 충분, 무료 티어로 커버 가능
→ 멀티플레이/랭킹/유저 데이터 영구 저장이 필요해지는 시점에야
  비로소 서버비 논의가 의미를 가짐 (현재 단계는 해당 없음)

이유 3 — 유입 동선과의 정합성
숏츠/카드뉴스 CTA("게임에서 확인 👉") → 링크 클릭 → 즉시 플레이
앱이면 "설치" 단계가 끼어들어 이탈 폭증 → 웹은 마찰 없음
```

### 8-2. 웹 아키텍처 — 무게 관리 원칙 (신규 확정)

```
원칙: EP가 쌓여도 사이트 전체는 가벼워야 한다

1) 에피소드별 완전 분리
   archiveoculus.com/play/ep01, /ep02 ... 각 EP = 독립 HTML/JS 묶음
   → 플레이 중인 EP의 리소스만 로드, 나머지 EP는 영향 없음

2) 미디어(이미지/음성) 별도 CDN 보관
   코드와 분리하여 저비용 CDN(Cloudflare R2, Supabase Storage 등)에 저장
   → 사이트 코드 자체는 항상 가볍게 유지, 재방문 시 캐싱으로 더 빨라짐

3) 세이브는 가볍게
   기본은 브라우저 로컬 저장, 클라우드 세이브/통계는
   구독자 전용 기능으로만 별도 경량 DB 사용

4) 목록(허브) 페이지는 압축 썸네일만 사용
   EP가 수십 개로 늘어도 허브 페이지 자체가 무거워지지 않도록 관리

→ "도서관이 무거워지는 게 아니라, 펼친 책 한 권의 무게만 체감된다" 원칙
```

### 8-3. 화면 방향 — 세로/가로 자동 대응 (신규 확정, 1차 수정)

```
결론: 어느 한쪽을 "메인"으로 고정하지 않고, 유저가 그 순간 들고 있는
      화면 방향을 그대로 따라간다 (기기 회전 자동 감지 + 레이아웃 전환)

수정 배경:
- 1차안은 "모바일=세로 고정"으로 단정했으나, 우리 콘텐츠 포트폴리오는
  가로(롱폼 유튜브)와 세로(숏츠/틱톡)가 함께 존재함을 반영하지 못한 결론이었음
- 롱폼에서 유입되는 유저는 시청 시점에 이미 가로 화면 모드인 경우가 많아
  (PC 가로, 모바일도 회전해서 풀스크린 시청) 가로 진입이 자연스러움
- 숏츠에서 유입되는 유저는 세로 그립 그대로 진입하는 게 자연스러움
- 따라서 "세로 강제"가 아니라 "기기 회전을 감지해 자동으로 레이아웃을
  전환"하는 반응형 설계가 원칙 — CSS 미디어 쿼리(orientation) 기반,
  구현 난이도는 낮음. 핵심 공수는 세로/가로 두 레이아웃을 각각
  보기 좋게 디자인하는 작업에 있음

레이아웃 분기:
세로형 (숏츠/모바일 세로 그립 유입 시 자동 적용)
  텍스트박스 풀폭 세로 스택 / 랜턴·스탯은 접이식 아이콘(탭하여 펼침)
  일러스트는 세로 비율 크롭 또는 정사각형(1:1) 위주로 제작

가로형 (롱폼/PC/모바일 가로 회전 시 자동 적용)
  기존 UI 시안 그대로 (랜턴+텍스트박스 나란히 배치, 일러스트 풀샷 활용)

제작 효율화 팁:
AI 이미지 생성 단계에서 정사각형(1:1)에 가깝게 뽑으면
세로/가로 양쪽 레이아웃에서 추가 크롭 작업 없이 무난하게 활용 가능

구현 방식:
CSS 미디어 쿼리(orientation: portrait / landscape)로 기기 회전을 감지해
레이아웃을 자동 전환. 강제 회전 유도나 안내 팝업 없이 무엇을 들고
있든 그 상태에 맞는 화면이 바로 보이도록 처리

체크리스트:
- 세로/가로 모두 완성도 있는 레이아웃으로 준비 (한쪽을 보조로 취급하지 않음)
- 터치 친화적 버튼 크기 (선택지 오탭 방지)
- 한 화면 내 텍스트+선택지 완결 (과도한 스크롤 지양)
- 가변 글자 크기로 작은 화면 가독성 확보
```
